## 1. 発明の名称

**Auracastブロードキャスト情報管理システム、情報収集方法、および地図表示方法**

---

## 2. 技術分野

本発明は、Bluetooth LE Audio規格に基づくAuracastブロードキャストの情報を収集・蓄積し、地図上に表示する技術に関する。特に、複数のクライアント端末から収集したAuracast送信機の情報をサーバで管理し、ユーザがAuracast送信機の位置およびコンテンツ情報を事前に把握できるシステムに関する。

---

## 3. 背景技術

### 3.1 Auracast技術の概要

Auracast（オーラキャスト）は、Bluetooth SIGが2022年に発表したBluetooth LE Audio規格の一部であり、ブロードキャスト音声機能を提供する。Auracastにより、1つの送信機から複数の受信機に対して同時に音声を送信することが可能となる。

主な用途として、以下が想定されている：
- 公共施設での音声案内（空港、駅、美術館等）
- 多言語対応の音声ガイド
- イベント会場での同時通訳
- 店舗での音声広告・案内

### 3.2 従来技術の課題

従来、Auracast送信機を利用するためには、ユーザが実際に送信機の通信範囲（通常10〜100m程度）に入り、Bluetooth対応デバイスでスキャンを実行する必要があった。

この従来技術には以下の課題が存在する：

1. **発見困難性**: ユーザは事前にAuracast送信機の存在を知ることができない
2. **情報不足**: 送信機の位置に到着するまで、提供されるコンテンツの内容が不明
3. **探索コスト**: 利用可能なAuracast送信機を探すために現地を移動する必要がある
4. **一時的イベント対応困難**: 展示会やスポーツイベント等での一時的な送信機の発見が困難

### 3.3 関連する先行技術

FM放送信号の強度を地図上に表示する技術（US 12,476,725 B2等）が存在するが、これはFM放送という広域通信を対象としており、近距離通信であるBluetooth LE Audio（Auracast）特有の課題を解決するものではない。

---

## 4. 発明が解決しようとする課題

本発明は、上記従来技術の課題を解決するため、以下の技術的課題を解決することを目的とする：

1. Auracast送信機の存在および位置を事前に把握可能とすること
2. 送信機が提供するコンテンツの情報を事前に確認可能とすること
3. 複数のユーザからの情報を統合し、信頼性の高いデータベースを構築すること
4. 送信機の現在の稼働状況（リアルタイム可用性）を把握可能とすること

---

## 5. 課題を解決するための手段

### 5.1 システム構成

本発明は、以下の構成要素からなるAuracastブロードキャスト情報管理システムを提供する：

#### 構成要素一覧

| 符号 | 構成要素 | 機能 |
|------|----------|------|
| 100 | クライアント端末 | Auracast送信機を検出し、情報をサーバに送信 |
| 110 | BASSスキャン部 | Bluetooth LE AudioのBASS規格に基づきスキャンを実行 |
| 120 | メタデータ抽出部 | スキャン結果からメタデータを抽出 |
| 130 | 位置情報取得部 | GPSまたはネットワーク測位により現在位置を取得 |
| 140 | 情報送信部 | 抽出した情報をサーバに送信 |
| 200 | サーバシステム | Auracast情報を蓄積・管理 |
| 210 | 情報受信部 | クライアント端末から情報を受信 |
| 220 | 信頼性評価部 | 複数端末からの情報に基づき信頼性スコアを算出 |
| 230 | データベース | Auracast送信機情報を格納 |
| 240 | 可用性監視部 | 送信機の稼働状況を監視 |
| 250 | ジオフェンス設定部 | 送信機位置に基づきジオフェンスを設定 |
| 260 | 通知制御部 | ユーザへの通知を制御 |
| 300 | 地図表示アプリケーション | 地図上にAuracast情報を表示 |
| 310 | 地図表示部 | 地図およびPOIを表示 |
| 320 | 状態表示部 | 送信機のアクティブ/非アクティブ状態を表示 |
| 330 | 詳細情報表示部 | 送信機の詳細メタデータを表示 |

### 5.2 データ構造

#### Auracast送信機情報（データベース格納形式）

| フィールド | 型 | 説明 |
|------------|-----|------|
| transmitter_id | String | 送信機の一意識別子 |
| broadcast_name | String | ブロードキャスト名（BASSより取得） |
| latitude | Float | 緯度 |
| longitude | Float | 経度 |
| context_type | Enum | コンテキストタイプ（案内、音楽、通訳等） |
| language | String | 言語コード（ISO 639-1） |
| audio_config | Object | 音声設定（コーデック、ビットレート等） |
| reliability_score | Float | 信頼性スコア（0.0〜1.0） |
| last_detected_at | Timestamp | 最終検出日時 |
| report_count | Integer | 報告端末数 |
| is_active | Boolean | アクティブ状態フラグ |

### 5.3 処理フロー

#### 5.3.1 情報収集フロー

```
1. クライアント端末(100)のBASSスキャン部(110)が、
   Bluetooth LE AudioのBASS規格に基づきAuracast送信機をスキャン

2. メタデータ抽出部(120)が、スキャン結果から以下を抽出：
   - Broadcast_Name
   - Language
   - Context_Type
   - Audio_Configuration
   - Streaming_Audio_Contexts

3. 位置情報取得部(130)が、GPS等により現在位置を取得

4. 情報送信部(140)が、抽出したメタデータおよび位置情報を
   サーバシステム(200)に送信
```

#### 5.3.2 サーバ処理フロー

```
1. 情報受信部(210)が、クライアント端末(100)から情報を受信

2. 信頼性評価部(220)が、以下の処理を実行：
   a. 既存データベースから同一送信機の情報を検索
   b. 新規情報の場合、初期信頼性スコアを設定
   c. 既存情報の場合、報告端末数を増加させ信頼性スコアを更新
   d. 位置情報の整合性を検証

3. データベース(230)に情報を格納または更新

4. 可用性監視部(240)が、定期的に以下を実行：
   a. 各送信機の最終検出日時を確認
   b. 所定時間（例：24時間）を超過した場合、is_activeをfalseに更新
```

#### 5.3.3 地図表示フロー

```
1. 地図表示アプリケーション(300)が、サーバシステム(200)から
   Auracast送信機情報を取得

2. 地図表示部(310)が、地図上に送信機の位置をPOIとして表示

3. 状態表示部(320)が、is_active状態に基づき表示を差別化：
   - アクティブ: カラーアイコン
   - 非アクティブ: グレーアイコン

4. ユーザがPOIをタップした場合、詳細情報表示部(330)が
   送信機のメタデータ（名称、言語、コンテンツ種別等）を表示
```

## 6. 発明の効果

本発明により、以下の効果が得られる：

### 6.1 技術的効果

1. **事前発見の実現**: ユーザはAuracast送信機の通信範囲外からでも、送信機の存在と位置を把握できる

2. **コンテンツ情報の事前確認**: BAP/BASSメタデータの活用により、送信機が提供するコンテンツの言語、種別等を事前に確認できる

3. **信頼性の高いデータベース**: クラウドソーシング型の情報収集と信頼性評価により、誤情報を排除した高品質なデータベースを構築できる

4. **リアルタイム可用性**: 可用性監視により、現在稼働中の送信機のみを把握できる

### 6.2 ユーザ体験の向上

1. 訪問先でのAuracastサービス利用可否を事前に確認可能
2. 多言語対応のコンテンツを効率的に発見可能
3. 不要な移動・探索コストの削減

## 7. 実施形態

### 7.1 第1実施形態：基本システム

図1に示すように、本実施形態のシステムは、クライアント端末100、サーバシステム200、および地図表示アプリケーション300から構成される。

クライアント端末100は、スマートフォン、タブレット、またはBluetooth LE Audio対応のウェアラブルデバイスであり、Auracast送信機を検出する機能を有する。

サーバシステム200は、クラウドサーバとして実装され、REST APIを介してクライアント端末100および地図表示アプリケーション300と通信する。

### 7.2 第2実施形態：ジオフェンシング連動

図2に示すように、本実施形態では、ジオフェンス設定部250および通知制御部260を追加する。

ジオフェンス設定部250は、データベース230に蓄積されたAuracast送信機の位置情報に基づき、各送信機を中心とした半径R（例：200m）の円形領域をジオフェンスとして設定する。

クライアント端末100の位置がジオフェンス領域に進入した場合、通知制御部260がプッシュ通知を送信し、ユーザに近くのAuracast送信機の存在を通知する。

### 7.3 第3実施形態：コンテンツカテゴリ検索

図3に示すように、本実施形態では、Context_Typeに基づくカテゴリ検索機能を追加する。

ユーザは地図表示アプリケーション300上で、以下のカテゴリを指定してフィルタリングできる：
- 音声案内（Announcements）
- 音楽（Media）
- 多言語通訳（Live Audio Sharing）
- 緊急放送（Emergency）

## 8. 特許請求の範囲

### 請求項1（システム）
Auracastブロードキャスト情報管理システムであって、
Bluetooth LE Audio規格のBASS（Broadcast Audio Scan Service）を用いてAuracast送信機をスキャンし、スキャン結果に含まれるメタデータおよび位置情報をサーバに送信するクライアント端末と、
前記クライアント端末から受信した情報を蓄積するデータベースを有するサーバシステムと、
前記サーバシステムから取得したAuracast送信機情報を地図上に表示する地図表示アプリケーションと、
を備えることを特徴とするAuracastブロードキャスト情報管理システム。

### 請求項2（信頼性評価）
前記サーバシステムは、
複数のクライアント端末から受信した同一のAuracast送信機に関する情報に基づいて信頼性スコアを算出する信頼性評価部を備え、
前記信頼性スコアが所定閾値以上の情報のみを前記データベースに蓄積することを特徴とする請求項1に記載のAuracastブロードキャスト情報管理システム。

### 請求項3（可用性監視）
前記サーバシステムは、
前記データベースに蓄積されたAuracast送信機情報の最終検出日時を監視し、現在時刻との差分が所定時間を超過した送信機を非アクティブ状態として更新する可用性監視部を備えることを特徴とする請求項1または2に記載のAuracastブロードキャスト情報管理システム。

### 請求項4（メタデータ）
前記クライアント端末がスキャンにより取得するメタデータは、
Broadcast_Name、Language、Context_Type、およびAudio_Configurationの少なくとも1つを含むことを特徴とする請求項1から3のいずれか一項に記載のAuracastブロードキャスト情報管理システム。

### 請求項5（状態表示）
前記地図表示アプリケーションは、
前記Auracast送信機の前記アクティブ状態または非アクティブ状態に応じて、地図上の表示態様を異ならせる状態表示部を備えることを特徴とする請求項3に記載のAuracastブロードキャスト情報管理システム。

### 請求項6（ジオフェンシング）
前記サーバシステムは、
前記データベースに蓄積されたAuracast送信機の位置情報に基づいてジオフェンス領域を設定するジオフェンス設定部と、
前記クライアント端末から受信した位置情報が前記ジオフェンス領域内に進入した場合に、当該Auracast送信機の情報を含む通知を前記クライアント端末に送信する通知制御部と、
を備えることを特徴とする請求項1から5のいずれか一項に記載のAuracastブロードキャスト情報管理システム。

### 請求項7（方法）
コンピュータが実行するAuracastブロードキャスト情報収集方法であって、
Bluetooth LE Audio規格のBASS（Broadcast Audio Scan Service）を用いてAuracast送信機をスキャンするステップと、
前記スキャンの結果からBroadcast_Name、Language、およびContext_Typeを含むメタデータを抽出するステップと、
現在位置情報を取得するステップと、
前記メタデータおよび前記位置情報をサーバに送信するステップと、
を含むことを特徴とするAuracastブロードキャスト情報収集方法。

### 請求項8（プログラム）
コンピュータに、請求項7に記載のAuracastブロードキャスト情報収集方法を実行させるためのプログラム。

## 10. 要約

【課題】Auracast送信機の存在および位置を事前に把握し、提供コンテンツの情報を確認可能とする。

【解決手段】クライアント端末100がBluetooth LE Audio規格のBASSを用いてAuracast送信機をスキャンし、メタデータおよび位置情報をサーバシステム200に送信する。サーバシステム200は複数端末からの情報に基づき信頼性を評価し、データベース230に蓄積する。地図表示アプリケーション300は、蓄積されたAuracast送信機情報を地図上にPOIとして表示し、アクティブ/非アクティブ状態を視覚的に区別して表示する。
